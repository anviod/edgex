# 南向采集通道回归验证测试方案

> **文档性质**：南向采集通道联机/压测回归方案（步骤与判定标准）。  
> **最后更新**：2026-07-04（回归复测执行 + 排期更新）  
> **相关文档**：[ScanEngine 重构方案](../TODO/ScanEngine重构方案.html) · [ScanEngine 测试报告](../TODO/ScanEngine重构测试报告.html) · [南向驱动测试报告](南向驱动测试报告.html) · [B1 采集间隔核验](b1_poll_interval_verification.html) · [Q3 Phase A–D 复测](q3_phase_abcd_verification_2026-07-04.html) · [影子设备设计测试](../edge/6. 影子设备设计_测试文档.html)

* BACnet
* OPC UA
* Modbus（TCP / RTU）
* 通道级隔离
* 设备级隔离
* 轮询压力测试
* 10ms 采集节点间隔验证
* 数据正确性验证
* 异常场景注入

目标是验证：

> ① 采集通道独立运行
> ② 单设备离线不影响同通道其他设备
> ③ 不同通道互不影响
> ④ 在线设备数据点位持续正确更新
> ⑤ 轮询间隔控制为 10ms（避免串流）
> ⑥ 在全量压力测试下系统稳定

> 现行架构已引入 **ShadowCore** 与快照层；采集调度内核为 **ScanEngine**（`deviceLoop` / `CollectionScheduler` 已下线）。以下测试项在不改动北向实现的前提下，验证调度隔离、快照一致性与 Commit 发布路径。

# 一、总体测试架构

```
                ┌──────────────────────────────────────────┐
                │            Edge Gateway                  │
                │  ScanEngine（10ms Tick 全局调度）         │
                │    ├ ExecutionLayer                      │
                │    │   ├ SerialQueue（Modbus/DLT645 等）  │
                │    │   └ ParallelExecutor（OPC UA/BACnet） │
                │    └ ConnectionController（设备连接态）     │
                └────────────────────┬─────────────────────┘
                                     │ dispatch → Driver.ReadPoints
         ┌───────────────────────────┼───────────────────────────┐
         │                           │                           │
   BACnet Channel              OPC UA Channel              Modbus Channel
   (Parallel 执行)              (Parallel 执行)              (Serial 硬隔离)
            ▼（采集完成）
   ShadowCore（原子快照） → Snapshot Bus（合并/优先级） → Points Commit（差量） → MQTT/OPC UA（复用）
```

> 架构详图见 [ScanEngine 重构方案](../TODO/ScanEngine重构方案.html) §1；影子设备写入闭环见 [影子设备设计测试](../edge/6. 影子设备设计_测试文档.html)。

### 强制架构要求（必须验证）

| 项目   | 必须满足                   |
| ---- | ---------------------- |
| 调度内核 | ScanEngine 10ms Tick，任务经 ExecutionLayer 分发 |
| 通道/协议 | SerialQueue 串行硬隔离；Parallel/Limited 背压受控 |
| 设备连接 | ConnectionController 独立超时/退避/半开探测 |
| 设备失败 | 不得 panic / 不得阻塞 ScanEngine Tick |
| 采集间隔 | 同通道串行设备之间间隔 ≥10ms（Hard Jitter） |
| 超时控制 | 单设备超时 ≤3s              |
| 数据隔离 | 不得出现设备数据串流             |
| 快照一致 | ShadowCore Snapshot Version 单调递增，组完成后递增 |
| 发布源唯一 | 北向发布仅由 Points Commit 触发         |

# 测试组 A：基础轮询验证

### A1 正常轮询验证

**步骤：**

1. 所有设备在线
2. 设置轮询周期 1s
3. 设置设备间隔 10ms
4. 运行 30 分钟
5. 记录每个采集组完成事件（GroupComplete）与对应 Snapshot Version++；统计 Commit 次数与点位变更次数。

**验证项：**

| 验证点    | 判定标准     |
| ------ | -------- |
| 轮询不丢设备 | 所有设备都被轮询 |
| 数据刷新   | 所有点位持续更新 |
| CPU稳定  | < 70%    |
| 无panic | 日志无异常    |
| 快照生成 | 每组完成均有 Version++ |
| Commit去抖 | Commit 次数 < 原始点位变更数 |

---

# 测试组 B：单设备离线隔离验证（关键）

### B1 单设备离线

步骤：

1. 正常运行
2. 手动关闭 1台 Modbus
3. 观察 5分钟

验证：

| 验证点     | 标准      |
| ------- | ------- |
| 该设备状态   | Offline |
| 同通道其他设备 | 正常轮询    |
| 不出现阻塞   | 轮询日志连续  |
| 数据无串流   | 不出现数据错写 |
| 快照一致性   | 离线设备不生成新快照，在线设备快照正常 |
| 北向源唯一   | 无点位直推，仅 Commit 触发发布 |

### B2 同时离线多个设备

* 3台 Modbus

# 测试组 D：10ms 间隔验证（防串流）

### 目标

验证：

> 任意两个设备采集时间间隔 ≥10ms（串行协议经 SerialQueue 硬隔离；并行协议受 Hard Jitter 约束）

> **已核验样本**：[B1 采集间隔核验报告](b1_poll_interval_verification.html)（Modbus TCP 三从站，2026-02-14 现场 PollStart Δt 均 ≥10ms）。本组全协议复测仍按下列步骤执行。

---

## 验证方法

**方法一（推荐，联机）**：在 Driver `ReadPoints` 入口或 ExecutionLayer 派发处打时间戳日志：

```
[PollStart] device=xxx channel=xxx ts=...
```

导出日志后验证：

```text
设备A 10:00:00.000
设备B 10:00:00.012
设备C 10:00:00.023
```

**方法二（辅助）**：`GET /diagnostics/scan-engine` 查看 `scan_lag_p95_ms`、`scan_miss_deadline_total`（稳态应为 0）；串行队列可通过 `[ScanEngineAdapter]` 设备注册/执行日志对照同一 Tick 内设备顺序。

要求：

```
Δt >= 10ms

并且：ShadowCore Snapshot Version 仅在组完成后递增，不因单点位更新而递增。
```

---

### 判定标准

| 项目           | 标准    |
| ------------ | ----- |
| 平均间隔         | ≥10ms |
| 无 0ms 连续触发   |       |
| 不出现RTU CRC错误 |       |
| 不出现TCP粘包异常   |       |

---

# 测试组 E：数据串流验证（关键）

### 目标

验证：

> 设备A的数据不会写入设备B
且仅在 Mapping Commit 回写 Points 后才触发北向发布。

---

## 方法

人为制造：

* 设备A返回固定值 111
* 设备B返回固定值 222

运行 10分钟

验证数据库：

```
deviceA = 111
deviceB = 222
```

若出现：

```
deviceA = 222
```

说明数据串流（严重BUG）

# 七、现行调度模型（ScanEngine，2026-07）

> **说明**：旧版 `deviceLoop` / `CollectionScheduler` 与「Channel goroutine + device loop sleep 10ms」并发模型**已废弃**（阶段 4 完全切换）。现行唯一采集调度内核如下；本方案验收应对照此模型，而非重新引入并发 `for all devices`。

```
ScanEngine（10ms Tick）
    ↓ dispatch（PriorityQueue + Hard Jitter）
ExecutionLayer
    ├ SerialQueueManager   → Modbus / DLT645 / Omron FINS 等（同通道串行，Δt≥10ms）
    ├ ParallelExecutor     → OPC UA / BACnet IP 等（背压 512 并发 / 1000 req/s）
    └ LimitedExecutor      → S7 / EtherNet/IP / Profinet / IEC104 等
    ↓ Driver.ReadPoints + ConnectionController（超时/退避/CB）
ShadowCore.applyCollectToShadow
    ↓
Snapshot Bus → Points Commit → 北向复用
```

**禁止**在同通道串行协议上采用无队列的全设备并发 `ReadPoints`，否则易出现串流、阻塞与雪崩。并行协议亦须受 ExecutionLayer 背压与 DriverCircuitBreaker 约束。
