# 南向采集 → 北向服务 双向通信测试报告

**测试日期**：2026-06-25  
**测试范围**：ShadowCore 统一数据面、Pipeline 扇出、北向反控下发  
**测试环境**：Windows 10，Go 1.x，`internal/core` 集成测试

---

## 1. 背景与目标

架构设计要求：

1. **南向读**：ScanEngine → ShadowCore → ShadowBridge → DataPipeline → 北向（MQTT/OPC UA/HTTP 等）
2. **北向写**：北向客户端 → `WritePoint` → 驱动写设备 → ShadowCore 同步 → Pipeline 扇出

本次补齐 **北向写入后 Shadow/Pipeline 同步**（原先 `WritePoint` 仅写驱动，影子与北向不会立即更新），并新增端到端集成测试验证双向链路。

## 4. 北向协议写入能力现状

| 协议 | 上报（读） | 反控（写） | 写入路径 |
|------|-----------|-----------|----------|
| **MQTT** | `Publish(v)` via Pipeline | `SubscribeTopic` → `handleWriteRequest` → `sb.WritePoint` | ✅ 已实现 |
| **OPC UA Server** | `Update(v)` via Pipeline | 客户端 Write → `sb.WritePoint` | ✅ 已实现 |
| **edgeOS(MQTT)** | `Publish(v)` | `handleWriteCommand` → `sb.WritePoint` | ✅ 已实现 |
| **edgeOS(NATS)** | `Publish(v)` | Request/Reply → `sb.WritePoint` | ✅ 已实现 |
| **HTTP** | 定时/批量上报 | 无订阅写通道 | ⚠️ 仅上报 |
| **Sparkplug B** | `Publish(v)` | 无写订阅 | ⚠️ 仅上报 |
| **REST API** | `/api/values/realtime` | `POST /api/write` → `cm.WritePoint` | ✅ 已实现 |

> 本次变更使所有经 `ChannelManager.WritePoint` 的北向写操作均同步 Shadow 并扇出 Pipeline，与南向采集数据同源。

## 5. 验证场景说明

### 5.1 南向 → 北向（读）

1. ScanEngine 采集驱动数据
2. `ShadowCore.WriteShadowDevice(source=scan_engine)`
3. ShadowBridge 订阅变更，推入 DataPipeline
4. NorthboundManager.handleValue 扇出至 MQTT/OPC UA/Sparkplug/edgeOS

### 5.2 北向 → 南向（写）

1. 北向客户端发送写命令（MQTT write topic / OPC UA Write / REST `/api/write`）
2. `ChannelManager.WritePoint` 校验 RW 权限
3. 驱动 `WritePoint` 下发至设备
4. `publishWrittenValue` 更新 Shadow（source=write_point）
5. ShadowBridge → Pipeline → 北向/UI 实时刷新

| 项目 | 状态 |
|------|------|
